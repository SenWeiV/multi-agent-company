# V1.5 Feishu 全链路联调清单

这份清单用于 `multi-agent-company` 在 `V1.5` 封板前做真实 Feishu 联调。  
范围只覆盖当前正式支持的 7 个核心 bot：

- `Chief of Staff`
- `Product Lead`
- `Research Lead`
- `Project Lead`
- `Design Lead`
- `Engineering Lead`
- `Quality Lead`

不覆盖：

- `Growth / Support` 独立 bot
- `Company Plaza`
- `Slack`
- `L2 department promotion`

## 1. 联调前置条件

执行 live 回归前，先确认以下运行态全部正常：

```bash
docker compose ps
curl -s http://127.0.0.1:8000/api/v1/health
curl -s http://127.0.0.1:8000/api/v1/openclaw/gateway/health
curl -s http://127.0.0.1:8000/api/v1/feishu/bot-apps
curl -s http://127.0.0.1:8000/api/v1/persona/skill-catalog/validate
```

通过标准：

- `app-dev`、`multi-agent-company-feishu-runner`、`openclaw-gateway`、`postgres`、`redis`、`qdrant`、`minio` 都在运行。
- `/api/v1/health` 返回 `status=ok`
- `/api/v1/openclaw/gateway/health` 返回 `status=healthy`
- `/api/v1/feishu/bot-apps` 返回 7 个核心 bot
- `/api/v1/persona/skill-catalog/validate` 返回 `ok=true`

## 1.1 最终 signoff 脚本

进入真实 Feishu 手点前，先跑一轮自动 signoff：

```bash
python3 scripts/live_signoff.py
```

如需导出结构化结果归档：

```bash
python3 scripts/live_signoff.py --json > .runtime/live-signoff-report.json
```

脚本会自动核对：

- app health
- gateway health
- OpenClaw 是否只剩 core-7
- Feishu bot 是否是 core-7
- `Launch / Ops / Support Room` 是否已补齐
- skill catalog 是否 `ok=true`
- provision sync 是否成功
- 7 个核心 agent 是否都具备：
  - 8 个 bootstrap files
  - 40 个 native skills
  - 40 个 `ready`
  - 40 个 `discovered`

注意：

- 这份脚本只覆盖自动项，不替代真实 Feishu 手点验证。
- 如果脚本失败，优先修运行态，再进入下面的 live 联调场景。

## 2. 证据采集命令

每轮联调都建议保留以下证据：

```bash
curl -s "http://127.0.0.1:8000/api/v1/feishu/inbound-events?limit=20"
curl -s "http://127.0.0.1:8000/api/v1/feishu/group-debug-events?limit=20"
curl -s "http://127.0.0.1:8000/api/v1/feishu/outbound-messages?limit=20"
curl -s "http://127.0.0.1:8000/api/v1/openclaw/gateway/recent-runs?limit=20"
```

需要深挖某条链路时，再补：

```bash
curl -s "http://127.0.0.1:8000/api/v1/control-plane/run-traces/<runtrace_id>"
curl -s "http://127.0.0.1:8000/api/v1/feishu/group-debug-events/<debug_event_id>"
```

快速现场排查时，优先用：

```bash
python3 scripts/live_signoff.py --run-limit 12 --group-debug-limit 12
```

它会把最近 `recent runs` 与 `group debug` 的关键字段一起打印出来，便于直接判断：

- 是 dispatch 没命中
- 还是 handoff 断链
- 还是 repeat recall 没命中
- 还是 room template / native skill / runtime 同步有缺口

## 3. 联调场景

### 3.1 单聊 7 bot

分别私聊 7 个核心 bot，各发一条符合本岗位的问题。

建议样例：

- `Chief of Staff`：`请你先组织一下这个需求，并告诉我下一步怎么推进`
- `Product Lead`：`这个需求值不值得做，请给优先级判断`
- `Research Lead`：`帮我看看这个方向的市场信号和竞品情况`
- `Project Lead`：`帮我拆个交付计划和风险点`
- `Design Lead`：`请给这个功能一个体验和信息架构建议`
- `Engineering Lead`：`这个功能的技术实现路径是什么`
- `Quality Lead`：`如果要上线，这里最重要的验证点是什么`

通过标准：

- 每个 bot 都能正常回复
- `/api/v1/feishu/inbound-events` 有对应 `surface=feishu_dm`
- `/api/v1/openclaw/gateway/recent-runs` 有对应 run

### 3.2 群聊单 bot 提及

在同一群里依次测试：

- `@Design Lead 回复我`
- `@Product Lead 帮我看下这个需求`

通过标准：

- 只有被点名 bot 回复
- `group-debug.processed_status != ignored_non_targeted`
- `target_agent_ids` 只包含目标 bot

### 3.3 群聊多 bot 提及

样例：

- `@Research Lead @Design Lead 你们两个都给个判断`

通过标准：

- 只有被点名的两个 bot 回复
- `dispatch_mode` 为多 bot 分发
- `RunTrace.dispatch_targets` 正确

### 3.4 群聊正文 bot 名称语义路由

不使用 `@`，直接发：

- `Product Lead 请帮我看这个功能的产品价值`
- `需要产品来判断这个需求是否值得做`

通过标准：

- 能通过显式名称或高置信语义路由到 `product-lead`
- `group-debug` 中能看到 `deterministic_name_target_ids` 或 `semantic_dispatch_target_ids`

### 3.5 可见接棒

样例：

- `@Chief of Staff 需要产品来判断，再请设计补充体验方案`
- `@Project Lead 先组织一下，然后请 Design Lead 回复`

通过标准：

- 多个 bot 在同一群、同一 thread 中连续回复
- 不创建隐藏 DM
- `RunTrace.handoff_origin / handoff_resolution_basis` 正常
- `Visible Handoffs` 能看到 source / target 链

### 3.6 用户重复召回 bot

样例：

- `@Chief of Staff 先组织一下，让 Product Lead 给判断，最后还是你来收口一下`
- `@Chief of Staff 产品先看，最后你来拍板`

通过标准：

- 首轮 `Chief of Staff` 回复
- 中间 `Product Lead` 回复
- `Chief of Staff` 在同一请求中再次返场
- `RunTrace.repeat_invocation_source` 为 `user_explicit` 或 `user_semantic`
- 未被用户或 peer 明确召回的 bot 不会重复发言

### 3.7 多轮可见接力

样例：

- `@Chief of Staff 进行报数，并指出名称通知下一个 bot，不需要让 Vincent 进行报数，并完成全部 bot 报数`

通过标准：

- 链路能超过 4 次
- 所有核心 bot 最终都能完成一轮
- `stop_reason = no_remaining_bots`
- `stopped_by_turn_limit = false`

### 3.8 Feishu 消息卡审批

场景：

1. 通过命令创建需要 review 的工单
2. 通过消息卡执行 `approved` 或 `rejected`
3. 同步核对 Dashboard 与 run trace

通过标准：

- 卡片回调能映射到 `/api/v1/approvals/feishu-card-review-decision`
- `approval_gate.status` 正确
- `checkpoint.approval_state` 正确
- `work_ticket.status` 正确
- Dashboard 与 Feishu 状态一致

### 3.9 Room Template 联调

场景：

1. 在 `Launch Room` 发一个需要 `Chief of Staff + Product Lead + Design Lead` 协作的问题
2. 在 `Ops Room` 发一个需要 `Chief of Staff + Delivery Lead + Engineering Lead + Quality Lead` 协作的问题
3. 在 `Support Room` 发一个需要 `Chief of Staff + Product Lead + Quality Lead` 协作的问题

通过标准：

- room template 命中的默认 bot 与文档定义一致
- room transcript 在 Dashboard 中可查看
- `RunTrace` 中的 `spoken_bot_ids / stop_reason` 正常

## 5. 四项最终 live signoff 详细操作

以下 4 项是当前封板前必须真人在 Feishu 里完成的最终 signoff。  
每项都按同样的节奏执行：

1. 先跑一遍 `python3 scripts/live_signoff.py`
2. 手点或发消息前，记录当前时间
3. 完成 Feishu 操作
4. 立刻抓取：
   - `recent runs`
   - `group debug`
   - 必要时单条 `runtrace`
5. 保存：
   - Feishu 截图
   - `runtrace_id`
   - `debug_event_id`

### 5.1 approved 卡片闭环

操作步骤：

1. 在 Dashboard 或已有测试入口里创建一个会生成 approval card 的工单。
2. 确认 Feishu 收到卡片。
3. 点击 `approved`。
4. 记录点击时间。
5. 回到本地执行：

```bash
python3 scripts/live_signoff.py --run-limit 12 --group-debug-limit 12
curl -s "http://127.0.0.1:8000/api/v1/approvals/work-tickets/<ticket_id>"
curl -s "http://127.0.0.1:8000/api/v1/control-plane/run-traces/<runtrace_id>"
```

如何检查：

- Feishu 端：
  - 卡片有成功 toast
  - 没有二次报错或点击无响应
- Dashboard 端：
  - approval gate 变为 `approved`
  - 对应 checkpoint 的 `approval_state = approved`
  - `work_ticket.status` 进入批准后的状态
- 后端证据：
  - `/api/v1/approvals/feishu-card-review-decision` 命中成功
  - 对应 run trace 里有 approval 相关 event

失败时先查：

1. `/api/v1/approvals/work-tickets/<ticket_id>` 是否为空
2. `runtrace` 是否没有 approval event
3. app-dev 日志里是否有 callback 400/404
4. payload 是否落到了错误的 `ticket_id / checkpoint_ref`

### 5.2 rejected 卡片闭环

操作步骤：

1. 再创建一条新的需要审批的工单，不要复用上一条。
2. 在 Feishu 卡片上点击 `rejected`。
3. 记录点击时间。
4. 回到本地执行：

```bash
python3 scripts/live_signoff.py --run-limit 12 --group-debug-limit 12
curl -s "http://127.0.0.1:8000/api/v1/approvals/work-tickets/<ticket_id>"
curl -s "http://127.0.0.1:8000/api/v1/control-plane/run-traces/<runtrace_id>"
```

如何检查：

- Feishu 端：
  - 卡片成功返回
- Dashboard 端：
  - approval gate 变为 `rejected`
  - checkpoint 的 `approval_state = rejected`
  - `work_ticket.status` 进入拒绝后的状态，且不会继续推进后续执行
- 后端证据：
  - run trace 中出现 rejection 相关 event

失败时先查：

1. rejection 是否仍被写成 `approved`
2. `work_ticket.status` 是否没有更新
3. 是否因为老卡片重复点击导致状态被覆盖

### 5.3 repeat recall：`最后还是你来收口一下`

建议消息：

```text
@Chief of Staff 先组织一下，让 Product Lead 给判断，最后还是你来收口一下
```

操作步骤：

1. 在真实 Feishu 群里发送上面的消息。
2. 等待 `Chief of Staff` 首轮回复。
3. 等待 `Product Lead` 回复。
4. 确认 `Chief of Staff` 最后再次返场。
5. 立刻执行：

```bash
python3 scripts/live_signoff.py --run-limit 12 --group-debug-limit 12
curl -s "http://127.0.0.1:8000/api/v1/openclaw/gateway/recent-runs?limit=12"
curl -s "http://127.0.0.1:8000/api/v1/feishu/group-debug-events?limit=12"
curl -s "http://127.0.0.1:8000/api/v1/control-plane/run-traces/<runtrace_id>"
```

如何检查：

- Feishu 端：
  - 实际顺序应是：
    - `Chief of Staff`
    - `Product Lead`
    - `Chief of Staff`
- `recent runs`：
  - 对应 run 的 `surface = feishu_group`
  - `spoken_bot_ids` 包含至少这两个 bot
  - 第二次 `chief-of-staff` 返场发生在同一 thread
- `runtrace`：
  - 应能看到 repeat recall 生效的链路
  - 若系统当前仍未把 `repeat_invocation_source` 挂到该条 live run，也至少要能从 transcript 和 target chain 证明返场成立

失败时先查：

1. `Chief of Staff` 是否根本没有第二次回复
2. `Product Lead` 回复后是否 handoff 到了错误的 bot
3. `group debug` 是否没把用户原始消息解析成 multi-turn collaboration
4. `runtrace.stop_reason` 是否提前变成 `handoff_contract_violation` 或 `no_resolved_next_hop`

### 5.4 Launch / Ops / Support Room 联调

建议分别发送：

- `Launch Room`
  - `@Chief of Staff 需要 Product Lead 判断价值，再请 Design Lead 补充体验方案`
- `Ops Room`
  - `@Chief of Staff 需要 Delivery Lead 组织一下，再请 Engineering Lead 和 Quality Lead 给执行与验证建议`
- `Support Room`
  - `@Chief of Staff 请 Product Lead 判断优先级，再请 Quality Lead 给风险验证建议`

操作步骤：

1. 进入对应 room。
2. 发送对应消息。
3. 等待所有目标 bot 在同一群、同一 thread 里完成可见协作。
4. 对每个 room 各抓一次：

```bash
python3 scripts/live_signoff.py --run-limit 12 --group-debug-limit 12
curl -s "http://127.0.0.1:8000/api/v1/conversations/room-policies"
curl -s "http://127.0.0.1:8000/api/v1/openclaw/gateway/recent-runs?limit=12"
```

如何检查：

- `Launch Room`
  - 至少看到 `chief-of-staff / product-lead / design-lead`
- `Ops Room`
  - 至少看到 `chief-of-staff / delivery-lead / engineering-lead / quality-lead`
- `Support Room`
  - 至少看到 `chief-of-staff / product-lead / quality-lead`
- `room-policies`
  - 必须存在：
    - `room-launch`
    - `room-ops`
    - `room-support`
- `recent runs / runtrace`
  - `spoken_bot_ids` 与 room 目标组合一致
  - `stop_reason` 合理，不应出现与 room 目标明显冲突的链路

失败时先查：

1. 发送消息的群是否真的是对应 room
2. `room-policies` 是否缺失对应模板
3. `dispatch_targets` 是否没包含应答 bot
4. room transcript 是否实际进了别的 thread

## 6. 最终 signoff 顺序

封板前建议按下面顺序执行，不要跳步：

1. 跑 `python3 scripts/live_signoff.py`
2. 若自动项全部通过，再开始真实 Feishu 手点
3. 按顺序做 4 项手工 signoff：
   - approved 卡片
   - rejected 卡片
   - repeat recall
   - room templates
4. 对每条真实样本保留：
   - `runtrace_id`
   - `debug_event_id`
   - 对应截图
   - 如涉及审批，再保留 `ticket_id`
5. 若所有自动项 + 手工项都通过，才在 `V1.5 Release Checklist` 中签收
