## 1. P0 — 基础层（其他模块的前置依赖）

### 1.1 WorkTicket 状态机

**来源**：A2A Task 状态机概念借鉴

**现状问题**：`WorkTicket.status` 只有 `"draft"`，没有 FSM，任务创建后无法追踪状态流转。

**目标**：扩展 `WorkTicket` 为完整的任务生命周期状态机。

**状态流**：

```
draft → submitted → working → review → completed
                  ↘ blocked → working
                  ↘ failed
                  ↘ canceled
```

**实现**：

- `app/company/models.py` — 新增 `WorkTicketStatus` 枚举，扩展 `WorkTicket` 加入状态转换方法
- `WorkTicketService` 实现状态转换逻辑（已在 `system-architecture-and-modules.md` 4.11 定义）
- `RunTrace` 事件绑定 ticket 状态变更
- Feishu handoff 的 `stop_reason` 从 ticket status 派生

**为什么是 P0**：停止规则、事件流、关系建模动态层都依赖任务状态机。

**预估**：1-2 天

### 1.3 飞书群聊会话命令（/new、/end、/status）

**来源**：飞书群聊实际使用中发现的会话生命周期管理问题

**现状问题**：

当前飞书群聊中的 OpenClaw agent session 按 `agent:{id}:feishu:group:{channel_id}` 绑定，即同一个群聊中所有消息共用同一个 gateway session。这导致：

1. **Session 历史无限累积** — 每一轮新话题的对话都追加到同一个 session，agent 会说"CEO 你已经重复了 5 次"
2. **上下文污染** — 前一个话题的讨论结果影响新话题的 agent 行为
3. **PHASE_PLAN 生成失败** — session 历史过长时，agent 把新消息当成旧话题的重复，不产出结构化协议块
4. **无法主动终止** — 用户无法强制结束一轮已经跑偏的讨论（必须等 20 轮跑完或手动清 session）

**三层关系说明**：

```
飞书群聊层（用户视角）        App 调度层               OpenClaw Gateway 层（agent 大脑）
┌──────────────────┐     ┌──────────────────┐     ┌──────────────────────┐
│ /new 开新话题     │ ──→ │ 新 work_ticket    │ ──→ │ 新 session（新上下文）│
│ 20 轮 handoff    │     │ 新 thread         │     │ 旧 session → memory  │
│ /end 强制结束     │ ──→ │ 总结 + 关闭       │ ──→ │ session 标记 closed  │
│ 继续对话          │     │ 同一 work_ticket  │     │ 同一 session 追加    │
└──────────────────┘     └──────────────────┘     └──────────────────────┘
```

**设计方案**：

| 命令 | 触发方式 | 行为 |
|------|---------|------|
| `/new [话题描述]` | 用户在群聊发送 | 1. 为所有 active agent 创建新 gateway session（session_key 带 topic_id）<br>2. 旧 session 内容摘要写入 agent memory<br>3. 创建新 work_ticket + thread<br>4. 20 轮计数从 0 开始<br>5. 用话题描述作为 intake 消息发给 chief-of-staff |
| `/end` | 用户在群聊发送 | 1. 立即停止当前 handoff 队列<br>2. Chief of Staff 生成一份讨论总结<br>3. 所有 agent 当前 session 标记为 closed<br>4. 下一条非命令消息自动视为 `/new` |
| `/status` | 用户在群聊发送 | 1. 返回当前话题、轮次进度（N/20）、各 agent 发言次数<br>2. 显示当前 handoff 队列中待处理的 agent<br>3. 显示当前活跃 session 的简要状态 |
| `/reset` | 用户在群聊发送 | 1. 硬清理：删除所有 agent 的 gateway session 文件<br>2. 重置所有 thread 和 work_ticket 状态<br>3. 等价于"全部从零开始" |

**实现路径**：

| 改动点 | 内容 |
|--------|------|
| `app/feishu/services.py:handle_payload()` | 消息处理入口增加命令解析层，识别 `/` 前缀 |
| `app/feishu/commands.py`（新建） | 命令解析器 + 各命令的 handler |
| `app/openclaw/services.py` | 新增 `rotate_session()` 方法：关闭旧 session、摘要写入 memory、创建新 session_key |
| `app/openclaw/services.py:get_session_binding()` | session_key 加入 topic_id 维度：`agent:{id}:feishu:group:{channel}:topic:{topic_id}` |
| `app/feishu/services.py:_orchestrate_visible_handoffs()` | 检查 `/end` 中断信号，提前退出 handoff 循环 |

**session_key 演进**：

```
# 当前（按 channel 绑定，历史无限累积）
agent:opc-chief-of-staff:feishu:group:oc_xxxxx

# 改进后（按 topic 绑定，/new 创建新 topic）
agent:opc-chief-of-staff:feishu:group:oc_xxxxx:topic:t_20260517_001
```

**预估**：3-5 天
**依赖**：无（可独立开发）

## 2. P1 — 通信层升级

### 2.1 结构化 Agent 回复

**来源**：A2A 结构化 Message 概念借鉴

**现状问题**：`HANDOFF:` / `TURN_COMPLETE:` 靠正则从自然语言回复中解析，agent 可能忘写、格式错误、写在正文中间。

**目标**：替代文本协议行，改为 JSON 尾部块。

**数据模型**：

```python
class AgentReplyEnvelope(BaseModel):
    visible_text: str                    # 给人看的正文
    handoff_targets: list[str] = []      # 下一个 agent 的 employee_id
    handoff_reason: str | None = None
    turn_complete: bool = True
    completion_confidence: float = 1.0   # 0-1，agent 对自己完成度的判断
    ticket_status_suggestion: str | None = None
    artifacts: list[str] = []
```

**实现路径**（推荐方案 A）：

- 保持 LLM 输出自然语言，但在 system prompt 中要求 JSON 尾部块
- `app/openclaw/services.py` — 修改 system prompt + 增强 `_parse_structured_agent_reply()`
- 服务端解析 JSON 块而非正则匹配文本行

**兼容性**：`_parse_structured_agent_reply()` in `app/openclaw/services.py:1367` 已经在做类似的事，只需增强解析逻辑和上游 prompt。

**预估**：2-3 天

### 2.2 停止规则引擎 Phase 1-2

**来源**：三信号融合停止模型

**现状问题**：

| 信号 | 实现 | 问题 |
|---|---|---|
| Agent 自声明 | `TURN_COMPLETE: yes/no` 文本行 | 不可靠 — agent 可能忘写 |
| 硬性 turn limit | `feishu_visible_handoff_turn_limit` | 太粗暴 — 不区分任务复杂度 |
| Contract violation | `handoff_contract_violation` | 只检测格式错误，不检测内容质量 |

**Phase 1 — ticket status 驱动停止**：

- ticket.status == `completed` / `failed` / `canceled` → 强制停止
- ticket.status == `review` → 只允许 quality-lead 和 chief-of-staff 继续
- ticket.status == `blocked` → 暂停 handoff 链

**Phase 2 — 结构化 agent 完成声明**：

```python
class AgentCompletionDeclaration(BaseModel):
    turn_complete: bool
    completion_confidence: float  # 0-1
    reason: str                   # "task_fulfilled" | "need_more_info" | "out_of_scope" | "delegated"
    remaining_work: list[str]
```

**实现**：新增 `app/orchestration/stop_evaluator.py`

**预估**：Phase 1 — 1 天（依赖 §1.1），Phase 2 — 1 天（依赖 §2.1）

### 2.4 Agent 即时确认回复（Quick ACK + 异步深度回复）

**来源**：飞书群聊体验优化 — 用户 @ agent 或 handoff 后长时间无反馈

**现状问题**：Gateway agent 做 agentic 推理（web search、工具调用）可能耗时 1-2 分钟，期间用户在飞书群里看不到任何响应，不知道 agent 是否收到消息、是否在处理。

**目标**：每个 agent 被 @ 或被 handoff 后，秒级返回一条简短的"理解确认"，再异步生成完整的深度回复。

**设计方案（混合方案：轻量 LLM ACK + 异步深度回复）**：

1. **ACK 阶段**（2-3 秒内）：
   - 用小模型（如 haiku）或低 token 限制的 prompt
   - Agent 用 1-2 句话表达对消息的理解和处理方向
   - 示例："收到，这是一个评测流程设计的需求，涉及方法论选型和架构设计。我先做 intake framing，稍后给出详细分析。"
   - 立即发送到飞书群聊

2. **深度回复阶段**（正常耗时）：
   - 使用完整 gateway 调用（带工具、带深度推理）
   - 完成后追加到群聊作为完整回复
   - 保持现有 HANDOFF/PHASE_PLAN 协议行能力

**实现路径**：

| 改动点 | 内容 |
|--------|------|
| `app/openclaw/services.py` | 新增 `_generate_quick_ack()` 方法，用轻量 prompt + 低 max_tokens 快速生成 ACK |
| `app/openclaw/services.py:invoke_agent()` | 在正式 gateway 调用之前先调 `_generate_quick_ack()` 并返回 ACK |
| `app/feishu/services.py` | 发送 ACK 消息后，再执行完整 `generate_reply()` |
| `app/feishu/services.py:_orchestrate_visible_handoffs()` | handoff 接棒场景：在 `dialogue_service.generate_reply()` 之前先发 ACK |
| `app/orchestration/phase_orchestrator.py` | Phase Discussion 场景：`_execute_phase()` 中每个 speaker turn 先发 ACK |

**ACK prompt 设计要点**：
- 不允许调用工具
- max_tokens 限制在 100-150
- prompt 中注入 source agent 的回复摘要 + 用户消息，让 ACK 不是死板模板
- 明确告知 agent 只需表达理解和处理方向，不需要给出实质内容

**预估**：2-3 天
**依赖**：Phase Discussion（已完成）、Gateway 超时修复（已完成）

### 2.5 Gateway Timeout → 心跳检测（长任务不截断）

**来源**：飞书群聊测试中 Engineering Lead 在 gateway agentic mode 下写代码 10 分钟被 600s timeout 截断

**现状问题**：

当前 gateway 调用使用硬 timeout（`OPENCLAW_GATEWAY_TIMEOUT_SECONDS=600`）。但 agent 在 agentic mode 下可能执行长时间工作（写代码、创建项目骨架、执行复杂检索），实际有效产出却被 timeout 截断。截断后 app 收到的是不完整的中间状态。

- Engineering Lead 实际创建了完整的 `eval-pipeline` 项目（pyproject.toml、adapters、benchmarks、storage），但 10 分钟 timeout 后被中断
- 截断时 gateway 返回 `stopReason=toolUse`（正在执行 tool 时被强制停止）
- 其他 agent 认为 Engineering Lead "无响应"，触发 sequential handoff queue 的无限回绕

**目标**：去掉硬 timeout，改用心跳检测 — 只要 agent 在持续工作就不中断。

**设计方案（A+B 组合）**：

**A 层 — 轻量文件心跳（每 2 分钟）**：

```python
# app/openclaw/services.py — 新增 _wait_with_heartbeat()
def _wait_with_heartbeat(self, session_key: str, max_idle_seconds: int = 300) -> bool:
    """
    替代硬 timeout。每 120s 检查 session JSONL 的 mtime：
    - mtime 在增长 → agent 在工作，继续等待
    - mtime 超过 max_idle_seconds 没变 → agent 可能卡住，触发 B 层检查
    - 总安全上限 30 分钟（防止真正的死循环）
    """
```

**B 层 — CoS 语义巡检（idle 触发）**：

当 A 层检测到某个 agent 超过 5 分钟无文件变化时：
1. 读取该 agent 的 workspace 最近文件变更
2. 让 Chief of Staff 用轻量 prompt 判断："engineering-lead 最后产出了 X，当前是否需要干预？"
3. 三种判断结果：
   - `still_working` → 继续等待，重置 idle 计时器
   - `stuck` → 终止 agent session，用已有产出继续流程
   - `needs_input` → 通知用户，agent 在等待外部输入（如 GPU 资源）

**Phase Discussion 集成**：

在 `PhaseOrchestrator._execute_phase()` 中，当 phase lead 的 gateway 调用进入长等待时：
- 不阻塞整个 orchestrator
- 其他无依赖的 phase 可以并行启动（未来增强）
- 心跳状态通过 trace event 上报，对 CEO 可见

**改动点**：

| 改动点 | 内容 |
|--------|------|
| `app/openclaw/services.py` | `_wait_with_heartbeat()` 替代 `urlopen(timeout=...)` |
| `app/openclaw/services.py` | 新增 `_check_agent_workspace_activity()` — 读取 workspace 文件 mtime |
| `app/feishu/services.py` | handoff 长等待时发送"XX 正在处理中..."状态消息到群聊 |
| `app/orchestration/phase_orchestrator.py` | phase lead 长任务时的等待逻辑 + trace event |
| `.env.example` | `OPENCLAW_GATEWAY_TIMEOUT_SECONDS` → `OPENCLAW_GATEWAY_MAX_IDLE_SECONDS`（语义变更） |

**预估**：3-5 天
**依赖**：Phase Discussion（已完成）、Quick ACK（§2.4，建议先做，心跳期间可以先发 ACK）

### 3.4 语义收敛检测（ConvergenceDetector）

**来源**：差异化方向 B

**目标**：在 PhaseOrchestrator 中，基于讨论内容判断阶段是否应该结束。

**实现**：

```python
# app/orchestration/convergence_detector.py（新建）
class ConvergenceDetector:
    def evaluate(self, turn_records: list[PhaseTurnRecord]) -> ConvergenceSignal:
        """
        分析讨论记录，返回收敛信号：
        - information_gain: 最近一轮相对前序的信息增量（0-1）
        - perspective_coverage: 已表达的视角占比（0-1）
        - consensus_indicator: 是否出现共识信号
        - recommendation: "continue" | "suggest_wrap_up" | "force_complete"
        """
```

**检测方法（轻量，不需要额外 LLM 调用）**：
- Jaccard 相似度：最近一轮 vs 前序所有轮的关键词重叠率
- 视角覆盖：阶段内所有 participants 是否已至少发言一次
- 长度衰减：回复长度连续递减 → 信息增量不足
- Action item 稳定：连续两轮的 action item 列表无变化

**与 PhaseOrchestrator 集成**：
- `_execute_phase()` 循环中，每轮结束后调用 `ConvergenceDetector.evaluate()`
- 返回 `suggest_wrap_up` → 在 lead 的 phase context 中注入提示："讨论已趋于收敛，建议总结并声明 PHASE_COMPLETE: yes"
- 返回 `force_complete` → 等同于 max_turns 到达，强制结束

**预估**：2-3 天
**依赖**：Phase Discussion（已完成）

## 5. P4 — 高级能力

### 5.1 停止规则 Phase 4 — 三信号融合

**来源**：三信号融合停止模型

```python
def evaluate_stop(
    ticket_status: str,
    entropy_signal: EntropySignal | None,
    agent_declaration: AgentCompletionDeclaration,
    turn_count: int,
    turn_limit: int,
) -> tuple[bool, str]:
    # 强制停止：ticket 终态 or turn limit
    if ticket_status in ("completed", "failed", "canceled"):
        return True, "ticket_terminal_state"
    if turn_count >= turn_limit:
        return True, "turn_limit_reached"

    # 高置信停止：agent 声明完成 + 高置信度
    if agent_declaration.turn_complete and agent_declaration.completion_confidence > 0.9:
        return True, "agent_high_confidence_complete"

    # 熵收敛停止
    if entropy_signal and should_stop_by_entropy(...):
        return True, "entropy_converged"

    return False, "continue"
```

## 9. 竞品对比与差异化定位

### 9.1 市场定位

```
                  专注"干活"                专注"模拟"
                      ↑                          ↑
                      |                          |
                   Paperclip                TheAgentCompany
                  (运营框架)                   (评测沙盒)

                      |                          |

OPC 定位：中间地带 — 模拟真实公司部门协作 + 实用价值
```

### 9.2 Paperclip 架构对比

| 维度 | Paperclip | OPC |
|------|-----------|-----|
| 技术栈 | Node.js + React + PostgreSQL | Python + FastAPI + Feishu |
| 组织模型 | 严格树形（`reportsTo` 单父节点） | 多关系类型（delegates_to / collaborates_with / escalates_to） |
| 通信模型 | 异步 issue 评论 + wakeup 队列，@mention 横向通信 | 同步多轮对话 + 阶段式讨论编排 |
| 多方讨论 | 无原生支持（每次 wake 单个 agent 独立处理） | PhaseOrchestrator 多轮多方实时讨论 |
| 停止规则 | budget / max_turns / approval / issue lifecycle | turn_limit / PHASE_COMPLETE / handoff contract |
| 插件系统 | 成熟（event bus + host services + tools + plugin DB） | 无 |
| 核心强项 | 运营基础设施（预算/审批/工作树/环境管理/session 持久化） | 讨论编排（阶段/角色/上下文传递/协议驱动） |

**Paperclip 的主要局限**：
- 通信必须挂在 issue 下，无独立通道
- 无广播/群组通信机制
- 无结构化 agent-to-agent request/response 原语
- @mention 是非类型化的自由文本
- 组织图是 advisory，不约束运行时路由

**OPC 已有但 Paperclip 缺少的**：
- 阶段式多方实时讨论（PhaseOrchestrator）
- 多关系类型的 CollaborationEdge（虽然运行时未激活）
- DISCUSS_WITH 结构化横向沟通协议
- Phase context carry-over（阶段间讨论摘要传递）

### 9.3 四个差异化方向评估

#### A. 横向沟通 + 关系图运行时激活 — 核心差异化，高优先级

**当前状态**：`COLLABORATION_EDGES`（15 条有向边）和 `ROUTING_RULES`（7 条路由规则）定义在 `app/company/bootstrap.py`，但运行时不参与编排决策。

**差异化关键不在于"定义关系图"，而在于"让关系图驱动运行时行为"**：
- PhaseOrchestrator 的 `_select_next_speaker()` 查询关系图调整优先级
- `parse_phase_plan()` 校验 PHASE_PLAN 是否符合关系图拓扑
- Phase context 中注入关系信息（"你和 research-lead 是 collaborates_with 关系"）
- DISCUSS_WITH 目标不在关系图上时降权或 warning

**实现路径**：增强现有 `PhaseOrchestrator` + 新建 `app/orchestration/relationship_resolver.py`

#### B. 语义化停止规则 — 真正的新东西，中高优先级

**当前所有产品（包括 Paperclip）都是机械式停止**：budget / max_rounds / approval / 协议信号。

**OPC 可以做的**：
1. **内容收敛检测**：讨论历史中新增信息量持续下降 → 建议停止
2. **视角覆盖度**：所有必要视角已表达（基于阶段 participants 列表）
3. **共识达成检测**：最后两轮回复语义高度相似且无分歧信号

**实现路径**：已规划的 `停止规则 Phase 3`（§4.1 轻量熵信号）+ 新增 `Phase 5` 阶段讨论专用收敛检测

#### C. 关系类型多样化 — 合理，但优先级低于 A

**现有 3 种类型已覆盖核心场景**。扩展为 CC/notification/informal 等更多类型有意义，但**当前瓶颈是已有类型未在运行时生效**，不是类型不够多。

**建议**：先做 A（激活现有关系图），再在此基础上扩展 §4.2 InformationFlowEdge。

#### D. 沟通频次和深度动态化 — 有价值，但是第二阶段

**"根据关系亲疏决定沟通轮数"**——这依赖 A（关系图运行时激活）+ B（语义停止）先到位。

**实现路径**：§5.2 关系建模 Phase 4（非正式网络）的 `affinity_score` → 影响 `max_turns` 分配。

### 9.4 与 Paperclip 的关系策略

**不建议在 Paperclip 之上构建，建议互补共存**：

1. **短期**：独立发展，Phase Discussion 是核心竞争力
2. **中期**：OPC 作为 Paperclip 的 `openclaw_gateway` adapter（Paperclip 代码中已有此 adapter 类型定义），让 Paperclip 用户可以接入 OPC 的讨论编排能力
3. **长期**：OPC 负责"如何讨论"（编排层），Paperclip 负责"如何运营"（基础设施层），形成互补
